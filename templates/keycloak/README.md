# Commons Keycloak

Cross-cutting Keycloak / OIDC adapter configuration. A single `auth.keycloak`
block drives two distinct rendered outputs consumed by multiple workloads.
This feature produces no Deployment or StatefulSet of its own.

## Outputs

| Output | Resource | Consumed by |
|---|---|---|
| `keycloak.json` | ConfigMap `kramerius-keycloak` | `kramerius-public`, `kramerius-curator` |
| Keycloak section in `configuration.properties` | Each workload's own ConfigMap | `kramerius-public`, `kramerius-curator`, all worker groups |

The generated Keycloak section is rendered by
`kramerius.configurationProperties.keycloakSection`, which is included from the
shared configuration builder when `includeKeycloak=true`.

## Example

### Input — values.yaml

```yaml
auth:
  keycloak:
    realm: "kramerius"
    authServerUrl: "https://auth.example.com/auth"
    sslRequired: external
    resource: "kramerius"
    secret: "s3cr3t"
    verifyTokenAudience: false
    confidentialPort: 0
    useResourceRoleMappings: false
    policyEnforcer: {}
```

### Generated — keycloak.json

Mounted at `/root/.kramerius4/keycloak.json` in `kramerius-public` and
`kramerius-curator`.

```json
{
  "realm": "kramerius",
  "auth-server-url": "https://auth.example.com/auth",
  "ssl-required": "external",
  "resource": "kramerius",
  "verify-token-audience": false,
  "credentials": {
    "secret": "s3cr3t"
  },
  "confidential-port": 0,
  "policy-enforcer": {}
}
```

When `useResourceRoleMappings: true`, the key
`use-resource-role-mappings: true` is emitted. When `policyEnforcer` is
non-empty, its content is rendered under `policy-enforcer`.

### Generated — configuration.properties Keycloak section

Rendered into the `configuration.properties` ConfigMap of workloads that call
`kramerius.configurationProperties.merged` with `includeKeycloak=true`.

```properties
# Keycloak
keycloak.tokenurl=https://auth.example.com/auth/realms/kramerius/protocol/openid-connect/token
keycloak.clientId=kramerius
keycloak.secret=s3cr3t
```

The token URL is constructed automatically from `authServerUrl` + `realm`.
`authServerUrl` is normalized by trimming a trailing slash before URL assembly.

### Admin-client — generated globals.js

`admin-client` runtime config is generated by chart helper
`kramerius.adminClient.globalsJs` from structured values in
`adminClient.config.*` plus networking hosts (`ingress.*` / `gatewayApi.*`).

Keycloak alignment still matters:

- `auth.keycloak.authServerUrl` and `auth.keycloak.realm` should match the same
  Keycloak realm used by the admin SPA.
- `adminClient.config.keycloakLoginType` controls SPA login behavior.
- The admin SPA client id is configured in the SPA runtime/global config and can
  differ from backend adapter `auth.keycloak.resource`.

## Behavior when unset

If `auth.keycloak.realm` is empty:
- `keycloak.json` is rendered as `{}`.
- Keycloak section in `configuration.properties` is omitted.

## Notes

- `realm` and `resource` must match names configured in Keycloak.
- `secret` is rendered in plaintext into ConfigMaps.
- For production, prefer External Secrets / Sealed Secrets and inject values at
  deploy time.
