# Process Manager

The Process Manager is the task scheduler and dispatcher for all background operations in Kramerius. It receives task requests from both the curator and the public instance, persists them in its own PostgreSQL database, and distributes them to available worker pods.

## Position in the Stack

```
Kramerius Curator  ──▶  [Process Manager]  ◀── this component
Kramerius Public   ──▶        │
                              ├── process-db (CNPG PostgreSQL, RW)
                              └── Worker Groups (HTTP callbacks)
```

The Process Manager is also exposed directly via ingress (optional, with optional OAuth2 protection) so operators can inspect and manage tasks through its own UI.

## PVCs / Volumes

| Mount path in pod | Volume source | Access mode | Purpose |
|---|---|---|---|
| `/usr/local/tomcat/conf/logging.properties` | `kramerius-tomcat-logging` ConfigMap | RO | Tomcat log config |
| `/usr/local/tomcat/logs` | `tomcat-logs` PVC | RW | Tomcat application logs — each pod gets its own PVC via `volumeClaimTemplates` |
| `/root/.kramerius4/javaagent.jar` | `javaagents` PVC/NFS | RO | Java agent JAR (optional; `javaagent.enabled`) |

The Process Manager does **not** mount the Akubra or import storages — task execution happens in the worker pods.

## Configuration

### configuration.properties

Uses only the `auth.keycloak` shared root (Keycloak token URL + client credentials). Unlike the Kramerius apps and workers, the Process Manager does **not** include Akubra or Solr sections — it has no direct interaction with object stores or search indexes. JDBC credentials are injected via environment variables from the CNPG bootstrap Secret, not through `configuration.properties`. Add any process-manager-specific extra keys in `processManager.config.configurationPropertiesExtra` if needed.

### Environment Variables

| Variable | Source | Description |
|---|---|---|
| `JAVA_OPTS` | `processManager.env.JAVA_OPTS` | JVM flags, defaults to `-Xms512M -Xmx2G` |
| `JDBC_URL` | chart-generated | `jdbc:postgresql://<cnpg.processManager.cluster.name>-rw:5432/process` |
| `JDBC_USERNAME` | CNPG bootstrap Secret | Read from `process-db-secret` key `username` |
| `JDBC_PASSWORD` | CNPG bootstrap Secret | Read from `process-db-secret` key `password` |
| `TZ` | `timezone-configmap` | Timezone (`Europe/Prague`) |

## Database

Uses a dedicated CNPG-managed PostgreSQL cluster named `process-db` (see [cnpg.md](cnpg.md)). The schema is managed by the Process Manager application itself on startup (Liquibase or equivalent).

Credentials come from the bootstrap Secret generated by the chart from `cnpg.processManager.password`.

## Worker Registration and Dispatch

Workers register with the Process Manager by advertising their in-cluster base URL:

```
http://<POD_NAME>.worker-<GROUP_NAME>.<NAMESPACE>.svc.cluster.local:8080/worker/api/
```

Each worker pod's `WORKER_BASE_URL` env var is set to this URL using the Downward API (`$(POD_NAME)`). The Process Manager calls back to this URL to deliver task payloads.

The Process Manager tracks:
- Which worker pods are registered and healthy
- Which tasks are queued, running, or finished
- Task history and logs in `process-db`

## Ingress Exposure

The Process Manager can be exposed via its own ingress host (`ingress.processManager.host`). This allows operators to:
- View the task queue and history
- Cancel or retry tasks
- Monitor worker health

OAuth2 proxy protection is optional (`ingress.processManager.oauthProtected`).

## Resource Requests / Limits

| | Request | Limit |
|---|---|---|
| CPU | `100m` | `500m` |
| Memory | `640Mi` | `3Gi` |

The Process Manager schedules and dispatches tasks but does not execute them — CPU usage is low. The 3Gi memory limit covers the JVM heap (`-Xmx2G`) plus ~1Gi for JVM overhead.

## Dependencies

| Component | Protocol | Purpose |
|---|---|---|
| **Kramerius Curator** | HTTP REST | Receives task submission requests |
| **Kramerius Public** | HTTP REST | Receives task submission requests |
| **process-db** | JDBC / PostgreSQL RW | Persists task queue and history |
| **Worker Groups** | HTTP callbacks | Dispatches task execution |
| **nginx-ingress** (optional) | HTTPS | Operator UI access |

## Notes

- Only one instance should run at a time (`replicas: 1`). Running multiple instances against the same `process-db` without leadership election would cause duplicate task dispatching.
- The Process Manager exposes a REST API that both the curator and the public instance use — `PROCESS_MANAGER_URL` (env var) and `processManagerHost` (in `configuration.properties`) are set automatically by the chart to match the in-cluster service name.
- Worker pods must be able to reach the Process Manager service for status callbacks, and the Process Manager must be able to reach individual worker pods by their stable pod DNS names.
